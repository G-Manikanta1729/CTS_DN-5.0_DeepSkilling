How does Spring Security know

- Read the Authorization Header.

- Validate the JWT.
- Extract the username.
- Authenticate the user.
- Allow only authenticated requests to continue.

After completing this exercise,

# What is a Filter?

## Definition

A Filter is a Java component that intercepts an HTTP request **before** it reaches the controller.

Instead of allowing every request directly into the application,

filters examine the request first.

A filter can

- Read headers
- Validate tokens
- Log requests
- Modify requests
- Reject requests
- Continue processing

---

Suppose an application has


Instead,

If the token is valid,

This makes the application

- Cleaner
- Easier to maintain
- More secure

# What is OncePerRequestFilter?

## Definition

`OncePerRequestFilter` is a Spring Security base class that ensures a filter executes **only once for each HTTP request**.

Instead of executing the filter multiple times during request forwarding or internal dispatching,

Spring guarantees that

```
One HTTP Request

↓

One Filter Execution
```

This makes it the preferred base class for implementing custom security filters such as JWT authentication.

---

# Why Do We Extend OncePerRequestFilter?

Suppose a request is internally forwarded.

Example

```
Client

↓

Controller A

↓

Forward

↓

Controller B
```

If we implement a normal servlet filter,

the filter might execute multiple times.

```
Request

↓

Filter

↓

Forward

↓

Filter Again

↓

Controller
```

This creates unnecessary processing.

Using

```java
extends OncePerRequestFilter
```

ensures

```
Request

↓

JWT Filter

↓

Controller

↓

Forward

↓

No Second Execution
```

The filter runs only once.

---

# Why Is This Important for JWT?

JWT validation is not a lightweight operation.

Each validation requires

- Reading Authorization Header
- Extracting JWT
- Verifying Signature
- Checking Expiration
- Extracting Username
- Creating Authentication Object

Running this process multiple times for the same request wastes CPU resources.

Therefore,

Spring recommends using

```
OncePerRequestFilter
```

for JWT authentication.

# JWT Validation

After extracting the JWT from the Authorization Header,

the next step is to determine whether the token can be trusted.

Having a JWT alone is **not sufficient**.

Anyone can send a string that looks like a JWT.

For example,

```
eyJhbGciOiJIUzI1NiJ9...
```

or

```
abcdef.xyz.123456
```

The server must verify that

- The token was created by our application.
- The token has not been modified.
- The token has not expired.
- The token belongs to a valid user.

This entire process is called **JWT Validation**.

---

# Why Do We Validate JWT?

Suppose an attacker manually creates a fake JWT.

```
Header

↓

HS256

Payload

↓

Username = admin

Signature

↓

Fake Signature
```

If the server simply reads the username,

the attacker gains administrator access.

Instead,

the server validates

```
Signature

↓

Valid?

↓

Yes

↓

Accept Token

↓

No

↓

Reject Token
```

JWT validation prevents unauthorized access.

---

# Steps Performed During JWT Validation

Whenever a request arrives,

```
↓

↓

↓

↓

↓

Authenticate User
```

Only if **all** steps succeed does the request continue.

```
↓
```

# Security Considerations

JWT validation must always verify

- Signature
- Expiration
- Secret Key
- Token Structure

Never trust a JWT simply because it looks valid.

Always verify it before using any information from the payload.

Also remember,

JWTs are **signed**, not encrypted.

Anyone can decode the Header and Payload.

Therefore,

never store

- Passwords
- Credit Card Numbers
- Aadhaar Numbers
- PINs
- Bank Account Details

inside JWT claims.

---

# Exception Handling During Validation

Several problems may occur while validating a JWT.

Examples include

- Invalid Signature
- Expired Token
- Malformed JWT
- Unsupported JWT
- Empty JWT

When any of these situations occur,

the filter should immediately reject the request.

Typical flow

```
JWT Received

↓

Validation Error

↓

Exception Thrown

↓

Catch Exception

↓

401 Unauthorized

↓

Stop Processing
```

This ensures that only valid, trusted, and unexpired JWTs are allowed to access protected REST APIs.

At the end of this stage, the JWT Authorization Filter has successfully verified the token, extracted the authenticated user's identity, and prepared the information required for Spring Security to create an authenticated session for the current request.

# Spring Security Authentication

After the JWT has been successfully validated,

Spring Security still does **not** automatically consider the user as authenticated.

The JWT contains the username,

but Spring Security needs an **Authentication Object**.

Until this object is created,

Spring Security assumes

```
Anonymous User
```

Therefore,

after validating the JWT,

our filter must create an Authentication object and store it inside the Spring Security Context.

Only then can the request access secured REST APIs.

# JwtAuthorizationFilter.java

Complete Code

```java
package com.cognizant.springlearn.security;

import io.jsonwebtoken.Claims;
import io.jsonwebtoken.Jwts;
import io.jsonwebtoken.security.Keys;
import jakarta.crypto.SecretKey;
import jakarta.servlet.FilterChain;
import jakarta.servlet.ServletException;
import jakarta.servlet.http.HttpServletRequest;
import jakarta.servlet.http.HttpServletResponse;
import org.springframework.security.authentication.UsernamePasswordAuthenticationToken;
import org.springframework.security.core.context.SecurityContextHolder;
import org.springframework.security.web.authentication.WebAuthenticationDetailsSource;
import org.springframework.web.filter.OncePerRequestFilter;

import java.io.IOException;
import java.nio.charset.StandardCharsets;
import java.util.Collections;

public class JwtAuthorizationFilter extends OncePerRequestFilter {

    private static final String SECRET =
            "secretkeysecretkeysecretkeysecretkey";

    @Override
    protected void doFilterInternal(
            HttpServletRequest request,
            HttpServletResponse response,
            FilterChain filterChain)
            throws ServletException, IOException {

        String header = request.getHeader("Authorization");

        if (header != null && header.startsWith("Bearer ")) {

            String token = header.substring(7);

            SecretKey key = Keys.hmacShaKeyFor(
                    SECRET.getBytes(StandardCharsets.UTF_8));

            Claims claims = Jwts.parser()
                    .verifyWith(key)
                    .build()
                    .parseSignedClaims(token)
                    .getPayload();

            String username = claims.getSubject();

            UsernamePasswordAuthenticationToken authentication =
                    new UsernamePasswordAuthenticationToken(
                            username,
                            null,
                            Collections.emptyList());

            authentication.setDetails(
                    new WebAuthenticationDetailsSource()
                            .buildDetails(request));

            SecurityContextHolder.getContext()
                    .setAuthentication(authentication);
        }

        filterChain.doFilter(request, response);
    }
}
```

# Best Practices

- Always validate the JWT signature before trusting its contents.
- Use the same secure SecretKey for generation and validation.
- Keep the filter lightweight and focused only on authentication.
- Always call `filterChain.doFilter()` after processing the request.
- Do not store passwords inside the Authentication object.
- Use HTTPS so JWTs are encrypted during transmission.
- Log authentication failures without exposing sensitive token details.

# Interview Questions

### 1. Why do we create a JWT Authorization Filter?

To intercept every request, validate the JWT, extract the authenticated user, and integrate JWT authentication with Spring Security.

---

### 2. Why does the filter extend `OncePerRequestFilter`?

To ensure the filter executes exactly once for every HTTP request.

---

### 3. What is the purpose of `doFilterInternal()`?

It contains the custom logic for reading, validating, and processing the JWT.

---

### 4. Why do we use `Claims`?

`Claims` represent the JWT payload and provide access to information such as the subject, issue time, and expiration time.

---

### 5. Why is `SecurityContextHolder` important?

It stores the authenticated user so that Spring Security and the application recognize the current request as authenticated.

---

### 6. What is `UsernamePasswordAuthenticationToken` used for in JWT?

It represents the authenticated user after the JWT has been successfully validated.

---

### 7. Why do we call `filterChain.doFilter()`?

To continue processing the request through the remaining filters and eventually reach the controller.

---

### 8. What happens if JWT validation fails?

The filter should reject the request, typically with **401 Unauthorized**, and the controller should never execute.

---

### 9. Why do we use the same SecretKey for validation?

Because the JWT signature was created with that key. Using a different key results in signature verification failure.

---

### 10. What is the main responsibility of the JWT Authorization Filter?

To bridge JWT authentication with Spring Security by validating the token and creating an authenticated `SecurityContext` for every valid request.

# Quick Revision

- `JwtAuthorizationFilter` intercepts every incoming request.
- It extends `OncePerRequestFilter` to execute only once per request.
- The filter reads the `Authorization` header.
- It removes the `"Bearer "` prefix and extracts the JWT.
- The JJWT parser validates the token's signature and expiration.
- `Claims` provide access to the JWT payload.
- The username is extracted from the `sub` claim.
- A `UsernamePasswordAuthenticationToken` is created.
- `SecurityContextHolder` stores the authenticated user.
- `filterChain.doFilter()` forwards the request to the remaining filters and finally to the controller.
- This exercise completes the **JWT Validation and Authentication** phase, enabling secured REST APIs to trust requests carrying a valid JWT.
