# Introduction

In the previous exercise (**Part 1 – Add Spring Security**), we successfully secured our REST APIs using Spring Security. Every request was authenticated using HTTP Basic Authentication.

However, there was still one limitation.

Spring Security automatically generated a temporary user account every time the application started.

For example,

```
Username

↓

user
```

Password

```
Random Password Generated by Spring
```

This generated password changes every time the application restarts.

Although this is useful for quickly testing Spring Security, it is not practical for real-world applications.

Enterprise applications need to define their own users and assign different permissions based on the responsibilities of those users.

This is where **Users** and **Roles** come into the picture.

In this exercise, we will create our own users inside the application and assign roles such as **USER** and **ADMIN**.

Instead of using Spring Boot's generated user, our application will authenticate users using the credentials that we explicitly configure.

---

# Exercise Objective

The objective of this exercise is to configure custom users and assign roles using Spring Security's **In-Memory Authentication**.

After completing this exercise,

- The application should no longer use Spring Boot's default generated user.
- Custom users should be created.
- Each user should have one or more roles.
- Spring Security should authenticate using the configured users.
- Only authenticated users should access the secured REST APIs.

# What is Role-Based Access Control (RBAC)?

## Definition

Role-Based Access Control (RBAC) is a security model in which permissions are assigned to roles instead of individual users.

# What is In-Memory Authentication?

## Definition

In-Memory Authentication is a type of authentication where user information is stored directly in the application's memory instead of an external database.

The users exist only while the application is running.

Once the application stops,

all user information stored in memory is lost.

When the application starts again,

Spring recreates the users from the configuration.

---

# Why Do We Use In-Memory Authentication?

When developing an application,

we usually want to verify that authentication is working before connecting to a database.

Instead of creating

- Database Tables
- Entity Classes
- Repository Classes
- SQL Queries

Spring Security allows us to define users directly inside the application.

This makes development much faster.

# What is PasswordEncoder?

## Definition

`PasswordEncoder` is a Spring Security interface responsible for encoding and verifying passwords.

Instead of storing plain text passwords,

Spring Security stores encoded passwords.

During login,

the entered password is encoded again and compared with the stored value.

---

# Why Do We Need Password Encoding?

Suppose passwords are stored directly.

```
Username

↓

admin

↓

Password

↓

admin123
```

If someone gains access to the database,

every password becomes visible.

This is a major security risk.

Instead,

passwords are stored in encoded form.

Example

```
admin123

↓

$2a$10$P1m...

```

Even if someone sees the encoded password,

they cannot determine the original password easily.

---

# What is BCryptPasswordEncoder?

## Definition

`BCryptPasswordEncoder` is the most commonly used implementation of `PasswordEncoder`.

It uses the BCrypt hashing algorithm.

BCrypt is

- Secure
- Slow enough to resist brute-force attacks
- Automatically generates a random salt
- Recommended by Spring Security

---

# Why BCrypt?

Suppose two users have the same password.

```
pwd

↓

BCrypt

↓

Hash A
```

Another user

```
pwd

↓

BCrypt

↓

Hash B
```

Even though both passwords are identical,

the generated hashes are different because BCrypt uses a random salt.

This makes password cracking significantly more difficult.

# SecurityConfig.java

This is the most important file in this exercise.

It contains

- User Creation
- Role Assignment
- Password Encoding
- HTTP Security Configuration

Complete Code

```java
package com.cognizant.springlearn.security;

import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.security.config.Customizer;
import org.springframework.security.config.annotation.web.builders.HttpSecurity;
import org.springframework.security.core.userdetails.User;
import org.springframework.security.core.userdetails.UserDetails;
import org.springframework.security.crypto.bcrypt.BCryptPasswordEncoder;
import org.springframework.security.crypto.password.PasswordEncoder;
import org.springframework.security.provisioning.InMemoryUserDetailsManager;
import org.springframework.security.web.SecurityFilterChain;

@Configuration
public class SecurityConfig {

    @Bean
    public InMemoryUserDetailsManager userDetailsService(PasswordEncoder passwordEncoder) {

        UserDetails user = User.builder()
                .username("user")
                .password(passwordEncoder.encode("pwd"))
                .roles("USER")
                .build();

        UserDetails admin = User.builder()
                .username("admin")
                .password(passwordEncoder.encode("pwd"))
                .roles("ADMIN")
                .build();

        return new InMemoryUserDetailsManager(user, admin);
    }

    @Bean
    public PasswordEncoder passwordEncoder() {

        return new BCryptPasswordEncoder();

    }

    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http)
            throws Exception {

        http
                .csrf(csrf -> csrf.disable())

                .authorizeHttpRequests(auth -> auth

                        .requestMatchers("/countries/**")
                        .hasAnyRole("USER", "ADMIN")

                        .anyRequest()
                        .authenticated()

                )

                .httpBasic(Customizer.withDefaults());

        return http.build();
    }
}
```

# Best Practices

- Never store passwords in plain text.
- Always use a strong password encoder such as BCrypt.
- Assign permissions through roles rather than individual users.
- Keep security configuration centralized in `SecurityConfig`.
- Follow the principle of least privilege by granting only the permissions required.
- Use meaningful role names such as `ADMIN`, `USER`, `MANAGER`, etc.
- Use In-Memory Authentication only for learning, testing, and prototypes.

---

# Common Mistakes

## 1. Storing Plain Text Passwords

Wrong

```java
.password("pwd")
```

Correct

```java
.password(passwordEncoder.encode("pwd"))
```

---

## 2. Forgetting PasswordEncoder Bean

Without

```java
PasswordEncoder
```

Spring Security cannot validate encoded passwords correctly.

---

## 3. Forgetting Roles

Wrong

```java
.username("user")
.password(...)
```

Correct

```java
.roles("USER")
```

Every authenticated user should have an appropriate role.

---

## 4. Confusing Roles and Authorities

Roles are high-level permission groups.

Authorities are individual permissions.

Example

```
ROLE_ADMIN

↓

READ_USERS

WRITE_USERS

DELETE_USERS
```

---

## 5. Using In-Memory Authentication in Production

In-Memory Authentication is intended for

- Learning
- Testing
- Small demos

Enterprise applications use

- MySQL
- PostgreSQL
- Oracle
- LDAP
- Active Directory

to store users securely.

---

### 10. Why is In-Memory Authentication not used in enterprise applications?

Because users are lost when the application stops, it is not scalable, and it does not support persistent user management. Enterprise applications use databases or directory services such as LDAP.

# Quick Revision

- In-Memory Authentication stores users inside application memory.
- `UserDetails` represents one authenticated user.
- `UserDetailsService` loads user information.
- `InMemoryUserDetailsManager` manages users in memory.
- `PasswordEncoder` securely encodes passwords.
- `BCryptPasswordEncoder` is the recommended password hashing algorithm.
- Roles group permissions and simplify authorization.
- `hasAnyRole()` restricts access based on roles.
- `SecurityFilterChain` applies security rules to every HTTP request.
- This exercise replaces Spring Boot's default generated user with custom users (`user` and `admin`) and assigns them the roles `USER` and `ADMIN`.
