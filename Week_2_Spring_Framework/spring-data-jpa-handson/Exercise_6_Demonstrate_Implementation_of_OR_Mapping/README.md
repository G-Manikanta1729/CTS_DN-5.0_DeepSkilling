# Introduction

Object-Relational Mapping (O/R Mapping or ORM) is one of the most important concepts in enterprise Java application development. It provides a mechanism to map Java objects to relational database tables, allowing developers to work with objects instead of writing complex SQL queries manually.

In traditional JDBC applications, developers must explicitly write SQL queries for inserting, updating, deleting, and retrieving records from the database. As applications grow, maintaining these SQL statements becomes difficult and increases the amount of boilerplate code.

Spring Data JPA, together with Hibernate, simplifies database interaction by automatically handling object persistence, relationship management, SQL generation, and transaction management. Developers can concentrate on implementing business logic while the framework manages communication with the database.

In this exercise, a payroll management database is used to demonstrate the three most common entity relationships in Object-Relational Mapping:

- **Many-to-One Relationship**
- **One-to-Many Relationship**
- **Many-to-Many Relationship**

These relationships are implemented using JPA annotations such as `@ManyToOne`, `@OneToMany`, `@ManyToMany`, `@JoinColumn`, `@JoinTable`, and `mappedBy`.

# What is Object-Relational Mapping?

Object-Relational Mapping (ORM) is a programming technique that establishes a connection between object-oriented programming languages and relational databases.

Instead of writing SQL queries manually, developers work with Java objects.

Hibernate automatically converts Java objects into database records and database records into Java objects.

Example:

```
Java Class

↓

Employee.java

↓

Hibernate

↓

employee Table
```

Similarly,

```
Java Object

↓

Employee Object

↓

Database Row
```

The conversion is performed automatically by Hibernate.

---

# Why Object-Relational Mapping?

Without ORM:

- Developers write SQL manually.
- Developers manage database connections.
- Developers convert ResultSet objects into Java objects.
- Developers handle transaction management manually.
- Large amounts of repetitive code are written.

With Spring Data JPA and Hibernate:

- SQL is generated automatically.
- Transactions are managed automatically.
- Relationships are maintained automatically.
- Objects are persisted automatically.
- Boilerplate code is significantly reduced.

# Performance Considerations

For enterprise applications,

developers should follow these best practices.

### Use LAZY loading whenever possible.

Avoid unnecessary object loading.

---

### Use EAGER loading only when required.

Loading large collections eagerly increases memory consumption.

---

### Keep Transactions Short

Long-running transactions reduce scalability.

---

### Use Repository Methods

Avoid writing unnecessary native SQL queries.

Spring Data JPA already provides optimized CRUD operations.

---

### Use Bidirectional Relationships Carefully

Avoid infinite recursion while printing entities.

Always design `toString()` methods carefully.

---

# Best Practices

- Keep entities lightweight.
- Separate business logic into service classes.
- Keep repositories free of business logic.
- Use constructor injection where possible.
- Keep transactions at the service layer.
- Prefer `Set` over `List` for Many-to-Many relationships.
- Enable SQL logging during development.
- Use meaningful entity and column names.
- Follow layered architecture.
- Avoid exposing entities directly through REST APIs in large applications.

## Database Connection Failed

**Error**

```
Communications link failure
```

**Cause**

MySQL Server is not running or incorrect database credentials.

**Solution**

- Start MySQL Server.
- Verify username and password.
- Check the JDBC URL.

---

## LazyInitializationException

**Cause**

Trying to access a lazily loaded collection after the Hibernate session is closed.

**Solution**

- Use `FetchType.EAGER` when appropriate.
- Access the collection inside a transactional method.

---

## Entity Not Found

**Cause**

Requested record does not exist.

**Solution**

Verify that sample data is present in the database before executing the application.
